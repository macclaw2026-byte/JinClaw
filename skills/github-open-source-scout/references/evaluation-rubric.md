# Evaluation Rubric

Use this rubric when deciding whether a GitHub project is worth attention.

## Core dimensions

### 1. Problem relevance
- Does it solve a real problem we have?
- Does it improve product selection, marketing research, browser/data extraction, analytics, or workflow reliability?

### 2. Practical leverage
- Could this save repeated work?
- Could it deepen evidence quality or analysis quality?
- Could it help us infer better product/market/sales signals?

### 3. Maintenance quality
- Is the repo active enough?
- Is the code understandable?
- Are docs adequate?
- Is the project clearly abandoned?

### 4. Integration cost
- How hard would it be to adapt or learn from?
- Is it lightweight enough to become a local reference or script pattern?
- Would it demand ongoing dependency burden?

### 5. Safety / sensitivity
- Does it touch credentials, browser auth, scraping boundaries, or risky automation?
- Would direct adoption require `skill-security-audit`?

## Recommendation classes

- **Ignore** — weak relevance or weak quality
- **Monitor** — interesting, but not worth acting on yet
- **Learn from** — good ideas/patterns, but better copied into local references/skills than installed
- **Evaluate for guarded install** — potentially useful direct tool, but requires audit
- **Promote to local skill work** — strong candidate to improve or create an in-house skill
