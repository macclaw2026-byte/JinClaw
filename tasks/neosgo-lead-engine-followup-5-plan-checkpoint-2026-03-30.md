## 1. 计划目标

在不突破安全边界的前提下，选择最适合完成以下目标的执行路径：
1. 对每个目标网站都把全部已支持抓取工具逐一运行
2. 保留每站每工具的原始结果证据
3. 从网站维度比较不同工具的抓取效果
4. 给出每个网站内各工具的对比结果与推荐
5. 形成可复跑、可审计的本地测评闭环

硬约束：
- 不绕过登录、验证码、拼图、人机校验
- 不引入新的高风险第三方黑盒抓取方案
- 不突破本地设备、数据、网络安全边界
- 结果必须基于本地可核验证据，而不是口头印象

## 3. 候选执行路径对比

### 方案 A：直接基于现有单网址脚本，手工逐个网站执行并人工汇总

做法：
- 不改代码结构
- 直接多次手工调用 `run_stack_bakeoff.py`
- 对每个网站分别输出 JSON
- 人工整理最终比较结论

优点：
- 最快开始
- 改动最小
- 风险低

缺点：
- 容易遗漏网站或工具
- 结果不够标准化
- 手工汇总成本高，后续复跑与扩展差
- 不利于沉淀为长期可用能力

安全判定：
- 安全
- 但工程化和可审计性不足

结论：
- 不选为主计划，可作为临时补充方法

---

### 方案 B：只做单工具 smoke test 扩展，用站点级分类替代全工具对比

做法：
- 继续沿用 `tools/site_smoke_test.py`
- 只增强站点分类和规则
- 不真正把 5 个工具全部逐站跑完

优点：
- 复用现有脚本最多
- 输出快

缺点：
- 与用户要求不符
- 无法回答“每个网站每个工具”的比较结果
- 只能得到站点粗分类，不是完整 bakeoff

安全判定：
- 安全
- 但目标覆盖明显不足

结论：
- 明确不选

---

### 方案 C：本地重建“多网站 × 五工具 × 统一评分”的 bakeoff 闭环（推荐）

做法：
1. 保留现有 5 工具栈定义作为核心能力
2. 新建或扩展批量 runner，输入目标网站列表
3. 对每个网站依次运行全部 5 个工具
4. 统一保存原始 JSON 结果
5. 增加网站维度评分/比较层，输出每站工具排名与优缺点说明
6. 输出正式报告，明确每站哪些工具最有效、哪些失败、为什么失败

优点：
- 与用户要求最吻合
- 证据链完整：网站列表 → 工具逐一执行 → 原始结果 → 比较评分 → 结论
- 可重复执行，可扩展到更多网站
- 完全符合 in-house capability rebuild 路径

缺点：
- 比手工做法多一些实现工作
- 必须接受某些网站在安全边界下会被拦截或效果差

安全判定：
- 安全
- 不依赖绕过或高风险外部工件

结论：
- 选为主计划

---

### 方案 D：引入新的外部抓取器 / 反检测方案，提高网站成功率

做法：
- 参考外部现成 SaaS、代理、反检测或绕过型方案
- 优先追求更多站点“抓到内容”

优点：
- 理论上可能提升部分站点结果

缺点：
- 超出当前安全边界
- 审计性和本地可控性差
- 当前任务并不需要为了完成 plan 阶段去引入新高风险依赖

安全判定：
- 不允许作为当前主路径

结论：
- 明确排除

## 4. 选定主计划

选定：**方案 C：本地重建“多网站 × 五工具 × 统一评分”的 bakeoff 闭环**

原因：
1. 最符合用户的原始要求：每个网站都要把所有工具一个一个试过
2. 最能回答“网站维度”的对比问题，而不是工具维度的零散观察
3. 最符合当前已知本地能力结构：现有 5 工具原型已经具备，可直接扩成批处理能力
4. 最符合安全与审计要求：不引入新黑盒，不绕过站点保护

## 5. execute 阶段建议拆解

### Phase E1：固定目标网站列表
目标：
- 明确本轮测评要覆盖哪些网站
- 避免执行中范围漂移

通过条件：
- 网站列表写入配置或代码中的统一位置

### Phase E2：批量执行器
目标：
- 新建或扩展本地 runner，使其支持多网站批量运行全部 5 个工具

通过条件：
- 每个网站都生成标准化结果
- 任一工具失败也不影响其它工具继续执行

### Phase E3：统一评分与比较
目标：
- 从网站维度对每个工具进行比较

建议比较维度：
- 请求是否成功
- 状态码 / 最终 URL
- 标题提取质量
- 正文长度与可读性
- 链接提取质量
- markdown / rendered text 质量
- 运行耗时
- 阻塞 / 风险信号
- 综合推荐

### Phase E4：正式报告
目标：
- 输出逐网站的工具比较表与结论

通过条件：
- 每个网站都有：原始执行结果 + 统一评分 + 工具排序 + 简短结论
