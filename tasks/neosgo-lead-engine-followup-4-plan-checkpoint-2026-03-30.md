## 3. 候选执行路径对比

### 方案 A：只修依赖 warning，然后立即出报告

做法：
- 仅处理 crawl4ai venv 中的依赖 warning
- 不补更多测试/报告结构
- 直接依据当前 smoke test 结果出排序结论

优点：
- 最快
- 风险最低

缺点：
- 只解决了一部分“工具问题”，没有建立完整交付路径
- 即使 warning 消失，Walmart / 1688 / Temu 深层可抓性也未被真正验证
- 报告说服力一般，容易显得只是一次性结论

安全判定：
- 安全
- 但对目标覆盖不足

结论：
- 不选为主计划

### 方案 B：继续逐站手工测试，积累观察后人工写报告

做法：
- 继续手工对四站做更多 crawl4ai 调用
- 根据人工观察补充结论
- 手动汇总成报告

优点：

- 不需要先搭更完整流程

缺点：

- 容易回到“零散观察”状态
- 不利于后续重复验证或继续修复

安全判定：
- 安全
- 但工程化价值偏低

结论：
- 不选为主计划，只保留为必要时的补充调查方式

### 方案 C：本地重建“修复 + 四站统一复测 + 排序报告”闭环（推荐）

做法：
1. 在现有 smoke test 基础上继续本地重建能力
2. 优先做低风险工具修复：清理 crawl4ai venv 的依赖 warning 或至少把 warning 处理为结构化健康信号
3. 用统一脚本复测四站公开页面表现
4. 按统一维度评分：可读性、字段丰富度、稳定性、阻塞强度、站点限制
5. 输出正式排序与报告

优点：
- 最贴合用户真实目标：不是只看一次结果，而是先修、再测、再评、再报
- 完全符合 in-house capability rebuild 思路
- 结果可重复验证，后续还能继续扩展到更多站点/页面
- 报告会有更强证据链

缺点：
- 比“一步出结论”稍慢
- 需要接受并如实报告某些站点在当前安全边界下就是抓不通

安全判定：
- 满足当前安全边界
- 不依赖绕过型抓取或高风险外部工件

结论：
- 选为主计划

## 4. 选定主计划

选定：**方案 C：本地重建“修复 + 四站统一复测 + 排序报告”闭环**

原因：

3. 最符合安全边界：不绕过站点限制，不引入黑盒外部方案

## 5. 执行分阶段拆解

### Phase P1：工具健康修复
目标：
- 清理或控制 crawl4ai venv 中的依赖 warning
- 明确 warning 是否影响结果判定

通过条件：
- warning 根因清楚
- 修复动作低风险、可回滚
- 若暂不修，也要把它纳入健康状态输出，不再隐式忽略

### Phase P2：统一四站复测
目标：
- 基于统一脚本再次复测四站
- 保证结果可复跑、可对比

通过条件：
- 每站至少有统一字段：return code / stdout_len / classification / warning_flags / evidence

### Phase P3：评测排序
目标：
- 建立明确评测维度并排序

建议维度：
- 页面可读性
- 可提取结构丰富度
- 当前稳定性
- 站点阻塞强度
- 后续扩展潜力

### Phase P4：正式报告
目标：
- 生成面向结果的简洁报告
- 报告中明确哪些是已修复、哪些仍受站点边界限制

通过条件：
- 报告包含排序、证据、限制、建议
