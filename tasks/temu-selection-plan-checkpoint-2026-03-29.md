# Temu 选品任务 plan 阶段检查点

日期：2026-03-29
任务来源：background runtime execution request / task_id `neosgo-lead-engine-followup-3`
阶段：plan

## 1. 计划目标

在不突破安全边界的前提下，对比多条可执行路径，选出最适合继续推进“稳定、可验证的 Temu 选品测试结果”的方案。

硬约束：
- 不突破登录/验证码/拼图边界
- 不引入高风险第三方执行工件
- 不用低质量、不可验证数据凑结果
- 保持输出可回查、可审计、字段级证据优先

## 2. 候选执行路径对比

### 方案 A：继续直接打通 1688 -> Temu 前台链路

做法：
- 继续尝试在 1688 和 Temu 前台公开页面上直接获得更完整商品级数据
- 依靠浏览器行为优化、少量人工观察、页面切换策略来提高成功率

优点：
- 路径最接近原目标设想
- 如果成功，字段天然贴近实际商品比对

缺点：
- 已验证存在稳定性问题：1688 登录门槛、Temu 安全验证
- 很容易把精力花在反复碰站点边界，而不是沉淀稳定证据管线
- 不符合“高置信优先”的当前推进原则

安全判定：
- 只要不绕过验证，本身可尝试
- 但作为主路径风险高、稳定性差，不应作为当前主计划

结论：不选为主计划，只保留为低频补充验证渠道

---

### 方案 B：引入外部现成 Temu 抓取器 / 绕过型 scraping 方案

做法：
- 参考公开博客、GitHub、插件、SaaS 抓取器等现成方案，直接复用其对 Temu 的采集能力
- 借助其验证码处理、代理、反检测等能力扩大抓取量

优点：
- 理论上能更快获得更多数据

缺点：
- 与当前安全边界直接冲突
- 外部执行路径可信度、数据质量、维护性与合规性都不稳定
- 会把系统建立在不可审计和高依赖外部工件的基础上

安全判定：
- 不满足当前任务的安全姿态与批准路径

结论：明确排除

---

### 方案 C：公开可访问替代货源 + Temu 公开参照 + 字段级证据化

做法：
- 不再把 1688/Temu 直连采集当主路径
- 改用可公开访问、可稳定读取的供应侧或目录型来源扩充候选池
- 用 Temu 可公开访问的方向性页面或已获取的公开参照作市场映射
- 所有候选项都绑定字段级 evidence record（来源、字段名、字段值、置信度、裁决说明）
- 输出时把高置信结果和待补证结果分层

优点：
- 与“公开可验证 / 高置信优先 / 不猜”高度一致
- 最容易形成稳定重复执行的本地能力
- 最适合后续落到 DuckDB / CSV / 报表的结构化证据链
- 能逐步积累更正式的测试结果，而不是反复卡在访问限制上

缺点：
- 短期内数量扩张可能慢于激进抓取路径
- 某些商品需要接受“方向已验证但货源/售价仍需补证”的中间态

安全判定：
- 满足当前安全边界
- 满足 in_house_capability_rebuild 的核心思路

结论：选为主计划

---

### 方案 D：只做方法论文档，不继续产出测试样本

做法：
- 暂停样本扩充，只保留设计、规则、模板与研究文档

优点：
- 风险最低

缺点：
- 无法逼近“稳定测试结果”目标
- 对当前任务帮助不足

安全判定：
- 安全，但目标收敛差

结论：不选

## 3. 选定主计划

选定：**方案 C：公开可访问替代货源 + Temu 公开参照 + 字段级证据化**

原因：
1. 它最符合当前用户目标的真实含义：稳定、可验证，而不是表面上“抓到更多页”
2. 它最符合当前安全边界：不碰登录、不碰验证码、不引入绕过方案
3. 它能直接沉淀为本地可维护能力，而不是依赖外部黑盒
4. 它能和已存在的系统设计、样本结果、evidence schema 自然衔接

## 4. 计划落地结构

### Phase P1：公开来源路线图
目标：
- 列出可公开访问、可复查、可持续的候选来源族群

输出要求：
- 每个来源记录：source_family、可访问性、字段覆盖、噪声风险、适配用途

### Phase P2：字段级 evidence 结构落地
目标：
- 以 `evidence-schema.md` 为基础，把 Temu 选品结果从“表格结果”升级为“字段有出处的证据记录”

输出要求：
- 至少覆盖：采购侧价格/链接/平台、市场侧参考价/平台、置信度、备注/裁决说明

### Phase P3：高置信候选扩样
目标：
- 先扩到一批更正式的高置信样本，而不是急着追 200+

输出要求：
- 明确区分：
  - 已双侧验证（更高优先）
  - 单侧成立但待补另一侧证据（次级）

### Phase P4：结果分层与正式测试输出
目标：
- 形成更接近正式测试结果的交付结构

输出要求：
- 主结果区：仅高置信、可公开复查项
- 待补证区：方向成立但仍需补链路的项目
- 风险注释区：清楚说明不可稳定自动化的原因
